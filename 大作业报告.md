## 一、项目背景与选题动机

普通劳动者和中小企业 HR 在遇到工资拖欠、违法辞退、工伤认定、劳动仲裁等问题时，往往难以快速找到准确的法条依据，也缺乏起草立案申请书等材料的能力；律师咨询成本高、响应慢，公开网页信息又分散且时效性不一。本人在整理劳动争议案例时多次遇到「知道有问题，但不知道具体对应哪一条法律」的困境，因此希望构建一个可本地部署、可引用依据、可辅助文书生成的劳动法智能顾问。

本项目采用 **RAG + QLoRA 微调 + Agent 工具调用** 的组合路线：对三部核心法规内的劳动法问题，先混合检索再交由经 DISC-Law 劳动法子集微调的 Qwen2.5-7B 生成回答，并在 Gradio 前端支持「基座 / 微调」一键对比；对需要实时政策、地区流程、文书模板或非劳动法问题，通过 LangChain Agent 调用 Tavily 联网搜索与本地 RAG 工具。系统仅提供信息参考与文书草稿，不构成正式法律意见。

## 二、系统架构设计

系统采用「**路由分流 + RAG 增强 + 双模型协作 + 可切换微调权重**」架构：Gradio 前端接收用户输入后，由本地 Qwen2.5-7B（4bit 推理）担任工作流路由器；劳动法常规问答走 FastAPI RAG 检索，再结合用户所选基座或 LoRA v2 权重流式生成；延伸场景走 DeepSeek Agent 工具链。知识库离线构建，LoRA 在 LLaMA Factory 中单独训练后挂载到同一基座，无需 merge 即可部署对比。

## 三、技术选型说明

劳动法场景既要**准确引用法条**，又要**贴近法律问答表达习惯**，还要能处理**实时政策与文书生成**。仅靠 RAG 能保证条文可追溯，但生成风格偏泛化；仅靠微调则法规更新需重训且易幻觉具体条文；仅靠 Agent 则本地法条检索不稳定。因此在 AutoDL 单卡资源下，最终采用「**RAG 保准确 + QLoRA 保领域表达 + Agent 补实时**」三层组合，并用同一 Qwen 基座贯通路由与 RAG 回答，降低部署复杂度。

### 3.1 技术方案对比

| 方案 | 优点 | 缺点 | 是否采用 |
| ---- | ---- | ---- | -------- |
| 纯 RAG | 法条引用准确、可解释、易更新知识库 | 回答偏模板化；无领域 SFT 时法条组织能力弱 | 作为核心组件 ✅ |
| 纯微调 | 法律问答风格统一、可学到条号引用习惯 | 法规更新需重训；易编造未收录条文；单卡全参微调成本高 | ❌ |
| RAG + 微调 | 检索保准确，微调改善表达与条号引用 | 工程链路更长；需构建训练数据与评测 | ✅（主路径） |
| RAG + 工具调用 | 可处理实时政策、文书、非劳动法问题 | Agent 延迟高；工具调用不稳定 | ✅（延伸路径） |

### 3.2 最终选型理由

1. **劳动法常规问答**：走 **RAG + QLoRA**。知识库仅 219 条法条级切片，适合高精度检索；v2 LoRA 在 DISC 劳动法子集上训练后，更倾向直接引用《劳动合同法》等具体条文，而非泛泛建议。
2. **实时政策、文书、非劳动法**：走 **Agent + Tavily + labor_law_rag**，避免让本地模型编造网页内容。
3. **基座选择 Qwen2.5-7B 而非课程示例 1.5B**：与前端 RAG 推理、路由质量一致；7B 在法条理解与长回答上明显优于 1.5B（课程早期 1.5B 试训 loss≈1.51，仅作流程验证）。
4. **不 merge LoRA**：采用 `PeftModel` 挂载 adapter，通过 `disable_adapter()` 在 UI 中对比基座与微调，节省约 15GB 磁盘且便于演示微调收益。

## 四、技术实现说明

### 4.1 RAG 模块

本项目的 RAG 不是「直接把 Word 切块丢进向量库」，而是经历 **原始导入 → 规则清洗 → 法条级重组 → 向量化 → 混合检索 → API 服务 → 前端注入 Prompt** 的完整离线/在线两阶段流程。核心脚本：`data/rag_law/chunking.py`（切片）、`step2_embedding.py`（入库）、`step3_search.py`（检索）、`rag_api.py`（服务）。

#### 4.1.1 数据来源与原始形态

| 项目 | 说明 |
| ---- | ---- |
| 法规范围 | 《中华人民共和国劳动合同法》《中华人民共和国劳动争议调解仲裁法》《工伤保险条例》 |
| 原始文件 | 3 份 Word 文档（`劳动合同法.docx` 等） |
| 中间缓存 | `data/rag_law/imported/knowledge.json` |
| 原始切片数 | **571** 条（导入工具按段落/行切分，粒度较碎） |
| 结构化后 | **219** 条法条级 Document（每「条」一条记录） |
| 各法规条数 | 劳动合同法 98 条；劳动争议调解仲裁法 54 条；工伤保险条例 67 条 |

`knowledge.json` 中每条原始 chunk 含 `content` 与 `metadata`（`source`、`position`、`document_id` 等），按 `position` 排序后送入切片器，保证章、节、条文的阅读顺序正确。

#### 4.1.2 清洗与法条级切片（`chunking.py`）

**设计原则**：法条是法律检索的最小语义单元，按「条」聚合比固定 `chunk_size` 更适合条号查询；法条之间**不做 overlap**，避免同一内容重复入库、干扰 rerank。

**清洗规则**（`_is_noise`）：

- 去掉纯空白、长度 ≤2 的碎片；
- 去掉「目录」「目次」等导航噪声；
- 章（`第X章`）、节（`第X节`）行只更新上下文，不单独入库。

**结构化流程**：

1. 按 `source` 分组，逐文件扫描有序 chunk；
2. 正则 `^(第…条)` 识别法条起始，将同一条下的续行合并为完整 `article_text`；
3. 从文件名规范化 `law_name`（如 `劳动合同法` → `中华人民共和国劳动合同法`）；
4. 从条文首句提取 `topic`（主题短语，最长 40 字）；
5. 生成统一 `page_content` 模板：

```text
法律名称：…
条号：第X条
章：第X章 …
主题：…
条文原文：
……
```

**输出 metadata 字段**：`law_name`、`article_no`、`chapter`、`section`、`topic`、`source`、`chunk_type=article`、`start_position`、`end_position`、`document_id`。

**迭代说明**：早期保留 571 条细碎片时，向量检索噪声大、条号问答易漏检；重构为 219 条法条级切片后，「劳动合同法第38条」类查询命中率明显提升。

| ------ | ---- |

#### 4.1.4 混合检索与重排（`step3_search.py`）

在线检索由 `semantic_search()` 统一入口，按优先级执行：

```
用户 query
    │
    ├─① 精确条号匹配（exact）
    │     解析「劳动合同法第三十条」→ law_name + article_no
    │     在 metadata 上精确过滤；命中则直接返回，跳过后续融合
    │
    └─② 混合候选召回（未精确命中时）
          ├─ 关键词检索（keyword）：内置 BM25 风格打分
          │    对 law_name / article_no / topic / content 加权分词
          │    含中文 2-gram，停用词过滤「怎么办」「如何」等
          ├─ 向量检索（vector）：Chroma similarity_search，候选约 12~24 条
          └─ RRF 融合：k=60，按 (law_name, article_no, source) 去重合并
                │
                ▼
          ③ BGE Reranker 重排（`bge-reranker-base`，CPU）
                exact 结果保留优先；其余按 cross-encoder 分数排序
                │
                ▼
          返回 top_k（API 默认 3，前端默认 5）
```

| 参数 | 值 | 说明 |
| ---- | --- | ---- |
| `CANDIDATE_K` | 10 | RRF 候选池大小 |
| `RERANK_RELEVANCE_THRESHOLD` | 0.3 | 前端判定「是否有有效检索」的 rerank 分数下限 |
| 精确命中 | 跳过 rerank 阈值 | `retrieval=exact` 一律视为有效证据 |

每条结果带回 `retrieval` 字段（`exact` / `keyword` / `vector` / `hybrid` / `hybrid+rerank`），前端参考资料卡片会展示检索方式与 rerank 分数。

#### 4.1.5 API 服务与前端联调

**RAG API**（`rag_api.py`，`run_api.sh` 启动，默认 `:8000`）：

- `GET /health`：返回向量库条数（当前 **219**）；
- `POST /search`：返回结构化 `results` 列表；
- `POST /ask`：在 search 基础上拼接 `context` 字符串，供大模型 Prompt 注入。

**前端链路**（`web/rag_client.py` → `web/workflow.py`）：

1. Qwen Planner 改写 query 后调用 `ask_rag_api`；
2. `has_relevant_evidence()` 过滤低分结果（rerank < 0.3 且非 exact）；
3. 将 `context` 填入 `build_answer_prompt()`，与 Qwen（基座或 LoRA v2）联合生成；
4. `format_used_sources()` 在回答下方渲染「参考资料」卡片。

**相关性过滤示例**：问「刑法故意伤害判几年」时，知识库无刑法，rerank 分数低于 0.3，系统返回「未检索到相关资料」而非强行编造。

### 4.2 微调模块

- **基础模型**：`Qwen2.5-7B-Instruct`（`/root/autodl-tmp/models/Qwen2.5-7B-Instruct`）
- **微调方式**：**QLoRA**（训练时 4bit 量化 + LoRA；推理时 4bit 基座 + PEFT 挂载 adapter）
- **LoRA 参数**：`r = 16`，`alpha = 32`，`lora_dropout = 0.05`，`lora_target = all`
- **训练框架**：LLaMA Factory（`day4/LLaMA-Factory`），配置见 `examples/disc_law_qwen7b_v2_lora_sft.yaml`
- **训练数据**：约 **9000** 条——`disc_law_labor_qa_v2`（4000）+ `disc_law_triplet_labor_qa_v2`（5000）+ `identity`；数据由 DISC-Law-SFT 原始 Pair/Triplet QA 经 `prepare_labor_subset.py` 按劳动法关键词筛取，脚本与说明见 `DISC-Law-SFT/README.md`
- **训练超参**：`num_train_epochs = 2`，`learning_rate = 3e-5`，`per_device_train_batch_size = 1`，`gradient_accumulation_steps = 16`，`cutoff_len = 2048`，`bf16 = true`
- **训练时长**：约 **45 分钟**（`train_runtime ≈ 2700s`，RTX 4090）
- **训练指标**：`train_loss = 0.437`，`eval_loss = 0.926`（5% val split）
- **产出路径**：`saves/Qwen2.5-7B-Instruct/lora/disc_law_qwen7b_v2/`（`adapter_model.safetensors` ≈ 155MB）
- **部署方式**：前端 `--lora-adapter-path` 加载 adapter；Gradio「回答模型」单选切换基座/微调，底层 `PeftModel.disable_adapter()`，**无需 merge**
- **消融说明**：v3 / v3.1（更严格数据清洗 + 不同超参）eval_loss 更高、20 题表现不如 v2，故生产与报告均以 **v2** 为准

**微调效果量化评估**（20 题对比评测，`eval/run_base_v2_compare.py`，`repetition_penalty=1.1`）：

| 指标 | 基座模型 | v2 LoRA | 说明 |
| ---- | -------- | ------- | ---- |
| 评测集 | `eval/disc_law_eval_20.json` | 同左 | 劳动法常见问答 |
| eval_loss（训练集） | — | **0.926** | 优于 v3（0.999）、v3.1（1.021） |
| 条号引用准确性 | 中等，偶有张冠李戴 | **较高** | 如 Q18「第19条」基座混入工时休假错误内容，v2 给出正确第十九条原文 |
| 回答风格 | 偏通用科普、步骤化 | **偏法律答复体例** | 如 Q2 辞退，v2 直接列《劳动合同法》第三十九条六项 |
| 复读/啰嗦 | 较少 | Q6、Q11、Q18 偶发 | 训练数据模板化导致，后续可加 repetition_penalty 或清洗 |
| RAG 联调后差异 | — | **部分题目趋同** | 检索上下文强约束时，基座与 LoRA 输出结构可能接近；无 RAG 时差异更明显 |

**典型对比示例（无 RAG，纯模型）**：

- Q2「公司可以随意辞退员工吗？」：基座给 6 条泛化原则；v2 直接引用第三十九条并说明违法解除需赔偿金。
- Q18「劳动合同法第19条主要内容是什么？」：基座将第十九条误述为「合同类型+工时休假」；v2 正确给出试用期期限规定全文。

### 5.3 评估总结

- **RAG 链路（01、03）**：混合检索 + rerank 对条号与场景问题准确率约 **90%**

- **延伸/文书（04–05）**：DeepSeek + Tavily 成功率约 **80%**，受 API 与搜索结果质量影响
- **Agent 通用（08–09）**：non_labor 转 web_search 后表现正常
- **整体**：10 条功能测试 9 条完全正确，1 条（基座模式 Q2 类条号题）为部分正确；微调价值在「无 RAG / 条号精确题」上更明显

## 六、遇到的问题与解决方法

| 问题描述 | 原因分析 | 解决方法 |
| -------- | -------- | -------- |
| RAG 经常检索不到或回答泛泛 | 早期切片过碎；prompt 过严 | 重构为 219 条法条级切片；混合检索 + rerank；放宽 answer prompt |
| 1.5B 试训效果差（loss≈1.51） | 数据仅 300 条、训练步数少、模型容量不足 | 改用 7B 基座；扩充至 9000 条劳动法子集；v2 训练 2 epoch |
| v3 / v3.1 微调不如 v2 | 数据清洗过严样本变少；v3 学习率偏大 | 保留 v2 为生产版本；v3 系列仅作消融记录 |
| UI 切换基座/LoRA 回答看起来一样 | RAG 上下文强约束 + `do_sample=False` 贪心解码 | 增加模型标识与终端日志；用条号题对比；离线 20 题评测验证差异 |
| 数据盘空间不足（>90%） | 重复 conda、旧 checkpoint、v1 adapter | 删除 v1 与 v3 中间 checkpoint；保留 v2 adapter 与基座 |
| bitsandbytes CUDA 库缺失 | 4bit 依赖 libnvJitLink | `run_qwen_frontend.py` 自动设置 `LD_LIBRARY_PATH` |
| LangChain Agent 启动报错 | LangChain 1.x API 变更 | 改用 `create_agent` + messages 格式 |

## 七、边界情况分析

### 7.1 系统表现差的情况

| 情况描述 | 典型案例 | 原因分析 |
| -------- | -------- | -------- |
| RAG 约束下微调差异被掩盖 | 「拖欠工资怎么办」基座与 LoRA 几乎同文 | 检索结果与 prompt 模板主导结构，贪心解码输出趋同 |
| LoRA 复读训练模板 | Q6 工伤材料、Q11 非全日制试用期 | DISC 数据中问答模板重复；未在推理端统一加 repetition_penalty |
| 路由器 intent 误判 | 「离婚财产」被判为 labor_question | Planner 仅依赖 prompt，无置信度二次校验 |
| 知识库覆盖不足 | 公司法、民法典相关问题 | 仅 3 部劳动法规 219 条；需 Agent 联网补充 |
| 基座条号幻觉 | 「劳动合同法第19条」 | 未微调时易将邻近章节内容混入；需 RAG 或 LoRA 约束 |

### 7.2 当前方案的局限性

1. **双模型异构**：Qwen 负责 RAG 与路由，DeepSeek 负责 Agent，风格与延迟不一致。
2. **微调未 merge**：当前为 adapter 挂载，依赖 PEFT；若需 vLLM 统一部署仍需 export merge（约 15GB+ 磁盘）。
3. **评测与生产路径不一致**：离线 20 题为纯模型问答；线上默认 RAG+微调，指标不可直接等同。
4. **工具调用不可控**：Agent 是否调用、调用顺序由模型自主决定，缺少强制策略。
5. **法律效力边界**：系统仅能提供参考，个案赔偿金额、胜诉判断仍有幻觉风险。

## 八、改进方向

| 改进方向 | 预期效果 | 实现难度 |
| -------- | -------- | -------- |
| 推理端对 LoRA 路径加 `repetition_penalty` | 减少 Q6/Q11/Q18 类复读 | 低 |

| 扩展知识库至地方性法规与典型案例 | 减少对联网依赖，提升地区问答 | 高 |
| v2 LoRA merge + vLLM 统一部署 Qwen | 单模型贯通 RAG 与 Agent，降低双模型成本 | 高 |
